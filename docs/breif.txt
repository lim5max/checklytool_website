1. Пользователь должен иметь возможность создавать новую "проверку"
2. Для каждой проверки пользователь должен иметь возможность указать количество вариантов контрольной работы.
3. Для каждого варианта пользователь должен иметь возможность либо загрузить изображение с эталонными ответами, либо ввести их вручную.
4. Пользователь должен иметь возможность указать критерии оценки (2-5) в процентах от правильного кол-ва ответов ученика
5. Пользователь должен иметь возможность использовать камеру устройства для сканирования и загрузки многостраничных работ учащихся.
5. Пользователь должен иметь возможность сразу посмотреть результат проверки в виде оценки, варианта, кол-ва правильных и не правильных ответов
6. В случае, если работа не была распознана, пользователь должен иметь возможность перезагрузить фотографию для этой конкретной работы.
7. Пользователь должен иметь возможность добавить фамилию и имя в карточке с проверкой если она не прочиталась с изображения

Система должна отправлять изображения работ и эталоны в API Openrouter (модель Gemini Flash) для распознавания и оценки.
Система должна отображать базовую статистику по результатам проверки (например, средний балл) для каждой работы
Пользователь должен иметь личный кабинет со списком всех прошлых проверок работ для доступа к их результатам.


Интерфейс сканирования должен предоставлять пользователю подсказки для получения качественных изображений.
Интерфейс на странице работы должен иметь кнопку с подсказками как и что делать
Система должна иметь механизм обработки сбоев и ошибок при обращении к внешнему API Openrouter
Приложение должно обеспечивать сохранность данных о проверках и результатах в личном кабинете пользователя


Общее видение UX: Интерфейс должен быть **современным, чистым и интуитивно понятным**. Основная цель — минимизировать количество шагов и когнитивную нагрузку на преподавателя. Кнопки и поля ввода должны быть большими и удобными для использования на мобильных устройствах. Взаимодействия должны сопровождаться приятными, не отвлекающими анимациями (реализованными с помощью **Framer Motion**) для создания ощущения отзывчивости.

### Ключевые принципы дизайна (Design Principles)
**Mobile First подход - так как люди будут пользоваться с телефона в основном**

Также:
1.  **Скорость как главная функция:** Каждое дизайнерское решение должно в первую очередь отвечать на вопрос: "Поможет ли это преподавателю сэкономить время?".
2.  **Ясность превыше всего:** Никаких двусмысленных иконок или сложных формулировок. Каждый шаг должен быть очевиден.
3.  **Минимум шагов:** Объединять действия там, где это возможно, чтобы сократить путь пользователя к цели.
4.  **Прогрессивное раскрытие:** Не перегружать пользователя информацией. Показывать только то, что нужно для текущего шага.
5.  **Доступность по умолчанию:** Следовать стандарту WCAG AA, чтобы приложением могли пользоваться все.
